# 2026-03-29 setup uv 多源与 trusted-host 优化设计

## 背景
当前安装链路中，`uv` 安装与 `uv sync` 的镜像策略不一致：

- `scripts/_common.sh` 中的 `pip` 安装链路已具备固定镜像回退能力，但源顺序不符合当前需求。
- `scripts/setup.sh` 中的 `uv sync` 仍采用“先探测可达镜像，再选择单一源执行”的模式，无法覆盖“探测通过但实际安装失败”的场景。
- 用户要求为所有源显式补充 host trust 参数，以降低 403 风险。

本次目标是统一两条链路的源策略与失败语义，确保只有在所有候选源都失败时才判定错误。

## 目标
1. 将 `uv` 安装链路中的 PyPI 源优先级统一为：**官方 → 阿里 → 清华**。
2. 将 `scripts/setup.sh` 中 `uv sync` 的镜像策略也统一为：**官方 → 阿里 → 清华**。
3. 对 `pip` 与 `uv sync` 的每次源尝试都附加对应 host 的 trust 参数。
4. 单个源失败时自动回退到下一个源，只有全部失败时才返回错误。
5. 保持现有安装失败摘要日志与非 PyPI 安装兜底（官方脚本 / conda / mamba / brew）语义不变。

## 非目标
1. 不调整 `uv` 安装链路在 PyPI 源全部失败后的后续兜底顺序。
2. 不扩展新的镜像源。
3. 不改动与本次需求无关的安装输出格式、交互流程或其他初始化步骤。

## 方案对比

### 方案 A（采用）
抽取统一的 PyPI 源定义，`pip` 安装与 `uv sync` 复用同一份顺序与 host 信息，按顺序逐个尝试。

**优点：**
- 两条链路行为一致，后续维护成本低。
- 能覆盖“可达性探测成功但安装失败”的场景。
- 更容易测试和验证。

**缺点：**
- 需要将 `setup.sh` 中的 `uv sync` 从“探测后单次执行”改为“按源循环尝试”。

### 方案 B（未采用）
仅分别修改两处逻辑的顺序与参数，不抽共用源定义。

**未采用原因：**
会继续保留两套实现，未来容易再次漂移。

### 方案 C（未采用）
保留预探测模式，只使用首个可达源，不做实际安装阶段的多次回退。

**未采用原因：**
不满足“全部源都尝试后才报错”的明确需求，稳健性不足。

## 设计细节

### 1. 统一源定义
在共享脚本中维护统一的 PyPI 源清单，每项包含：
- label
- index-url
- host

顺序固定为：
1. `pypi` → `https://pypi.org/simple`
2. `aliyun` → `https://mirrors.aliyun.com/pypi/simple/`
3. `tuna` → `https://pypi.tuna.tsinghua.edu.cn/simple/`

这份定义将作为 `pip` 与 `uv sync` 的唯一来源。

### 2. `pip` 安装链路
`_common.sh` 中的以下步骤统一改为按源顺序循环尝试：
- `pip install --upgrade pip --user`
- `pip install uv --user ...`

每次尝试都附带：
- `-i <index-url>`
- `--trusted-host <host>`

执行规则：
- 某个源成功则立即返回成功。
- 某个源失败则记录失败摘要日志并继续尝试下一个源。
- 全部源失败后，该步骤才返回失败。

### 3. `uv sync` 链路
`scripts/setup.sh` 中当前“先探测镜像可达性，再决定单个源”的逻辑改为按统一源清单直接尝试执行 `uv sync`。

执行规则：
- 先使用官方源执行 `uv sync`。
- 若失败，则依次回退到阿里、清华。
- 每次尝试都带对应 host 的 trust 参数。
- 只有全部源失败后，才输出“依赖安装失败”并返回错误。

这样可覆盖以下场景：
- 镜像探测成功但真实安装失败。
- 某镜像瞬时异常但后续镜像可用。

### 4. 失败语义
- **单个源失败**：属于可回退失败，记录日志后继续。
- **PyPI 源全部失败**：当前步骤失败。
- **`uv` 安装步骤整体失败**：在 PyPI 源全部失败后，继续沿用现有官方脚本 / conda / mamba / brew 兜底流程。
- **`uv sync` 失败**：PyPI 源全部失败后直接返回错误，不新增额外兜底机制。

## 影响范围
- `scripts/_common.sh`
  - 调整统一源定义顺序。
  - 复用或扩展多源尝试函数，使其可服务于 `uv sync`。
- `scripts/setup.sh`
  - 删除现有 `curl` 镜像可达性探测分支。
  - 改为基于统一源清单的 `uv sync` 顺序回退执行。

## 测试考虑
至少覆盖以下验证点：
1. 官方源成功时，`pip`/`uv sync` 不应继续尝试后续源。
2. 官方失败、阿里成功时，应正确回退并成功结束。
3. 前两个源失败、清华成功时，应最终成功。
4. 全部源失败时，应返回错误且保留失败日志。
5. `uv` 安装在 PyPI 源全失败后，仍应进入既有兜底安装路径。
6. `uv sync` 不再依赖预先 `curl` 探测结果。

## 风险与控制
- **风险：** `uv sync` 的 trust 参数写法若与当前 `uv` 版本不兼容，可能导致命令直接失败。
  **控制：** 实现后通过本地命令验证参数是否被正确接受。
- **风险：** 源定义重复或分散可能导致后续再次不一致。
  **控制：** 将顺序与 host 信息集中在共享脚本中维护。

## 实施后验收标准
1. `uv` 安装链路按“官方 → 阿里 → 清华”顺序尝试 PyPI 源。
2. `uv sync` 也按“官方 → 阿里 → 清华”顺序尝试。
3. 两条链路的每次尝试均显式传入 host trust 参数。
4. 单源失败不立即报错，全部源失败才报错。
5. 现有非 PyPI 兜底安装路径不被破坏。
