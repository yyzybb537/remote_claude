# 连接稳定性扫描修复与日志分流轮转设计

## 背景

当前代码库在远程连接、CLI 参数处理、脚本路径解析和异常边界上已经有多轮修复，但相关逻辑仍分散在 `remote_claude.py`、`server/server.py`、`client/*.py`、`bin/*` 与 `lark_client/*` 中。用户本次要求聚焦两类问题：

1. 扫描并修复 `server/`、`client/`、`bin/`、`remote_claude.py` 中可能存在的连接处理错误、参数处理错误、路径处理错误、异常未处理与保活处理异常。
2. 将当前分散的服务日志拆分为客户端 / 服务端 / lark 三类主日志，统一落到 `/tmp/remote-claude/`，并增加日志轮转能力。

现状里日志路径和职责并不统一：`server/server.py` 既写 `/tmp/remote-claude/...` 下的会话级调试日志，也仍保留 `~/.remote-claude/startup.log`、`~/.remote-claude/server.error.log` 等主日志路径；客户端与 lark 端也缺少统一的角色级主日志初始化入口。这会增加排障成本，也让后续修复很难判断应该在哪一层补日志。

## 目标

- 系统性扫描并修复以下类型的问题：
  - 连接处理错误
  - 参数处理错误
  - 路径处理错误
  - 异常未处理
  - 保活处理异常
- 在不改变现有协议语义和用户命令语义的前提下，允许做中等规模代码整理。
- 引入统一日志初始化模块，按角色拆分三类主日志：
  - `client.log`
  - `server.log`
  - `lark.log`
- 三类主日志统一写入 `/tmp/remote-claude/`。
- 主日志统一接入按大小轮转，采用用户确认的配置：`10MB × 5`。
- 保留现有会话级调试日志能力，例如 `_screen.log`、`_pty_raw.log`、会话级 debug 日志，不把这些调试产物错误合并进主日志。
- 同步更新 `CLAUDE.md` 与 `tests/TEST_PLAN.md`，补足与本次行为变更对应的测试说明。

## 非目标

- 不修改消息协议格式。
- 不新增面向用户的新 CLI 子命令。
- 不对整个 `argparse` 架构做无关重写。
- 不顺手重构与本次目标无关的展示逻辑、飞书卡片逻辑或解析器架构。
- 不引入复杂的自定义心跳协议；保活问题仅做必要修复。

## 用户确认的约束

- 采用“中度整理”策略：允许缺陷修复基础上的小范围结构收敛，但不做大范围重构。
- 日志方案采用“方案 A”：新增统一日志初始化模块，各端按角色获取 logger。
- 主日志轮转参数固定为：`10MB × 5`。

## 方案概述

本次采用方案 A，分两层推进：

1. **稳定性修复层**：扫描并修复 `server/`、`client/`、`bin/`、`remote_claude.py` 中与连接、参数、路径、异常、保活相关的真实缺陷。
2. **日志收敛层**：新增统一日志初始化模块，将客户端 / 服务端 / lark 三类主日志统一落到 `/tmp/remote-claude/` 并接入轮转，同时保留现有会话级调试日志。

执行顺序上采用：**稳定性扫描与问题清单 → 缺陷修复 → 日志收敛 → 测试补齐 → 文档同步**。

## 详细设计

### 一、变更边界与模块落点

#### 涉及范围

- `remote_claude.py`
- `server/server.py`
- `server/ws_handler.py`
- `client/base_client.py`
- `client/remote_client.py`
- `bin/*`
- `lark_client/*` 中与日志初始化直接相关的入口文件
- `tests/*` 中与上述行为对应的测试文件
- `CLAUDE.md`
- `tests/TEST_PLAN.md`

#### 设计边界

- 只处理本次明确要求的稳定性问题类型。
- 允许将重复的日志初始化、路径收敛、参数边界校验提取为复用逻辑。
- 不改变现有用户可感知命令集合，不要求用户调整现有命令写法。

### 二、稳定性扫描与修复策略

#### 1. 连接处理

重点检查以下链路：

- 本地 Unix Socket 客户端连接建立 / 关闭 / 异常返回
- 远程 WebSocket 建连、断连、读写异常透传
- 服务端客户端连接生命周期管理
- 远端控制命令（如 token / kill / lark）使用的短连接控制链路

修复原则：

- 建连失败、发送失败、读取失败、远端关闭要能被区分。
- 半断开状态下 `_connected`、`running`、连接对象引用应保持一致，避免出现“逻辑已断但对象未清”的状态。
- 不吞掉真正影响排障的边界异常；用户侧给简洁提示，细节进日志。

#### 2. 参数处理

重点检查以下内容：

- `remote_claude.py` 中远程命令的 `--remote / --host / --port / --token / session` 组合校验
- `attach`、`list`、`kill`、`token`、`regenerate-token`、`lark start/stop/status` 等远程路径
- help 模式下是否错误触发校验或副作用
- `bin/*` 入口脚本传参与 lazy init 后参数透传是否稳定

修复原则：

- 保持现有兼容写法，不改变用户命令语义。
- 解析、缺参报错、帮助输出行为尽量统一。
- 避免命令在 help 模式或参数不完整时提前触发真实连接、路径检查或副作用逻辑。

#### 3. 路径处理

重点检查以下内容：

- `/tmp/remote-claude/` 目录创建与写入
- socket 路径与 session 路径映射
- `bin/*` 与 `scripts/*` 在 symlink / 任意 cwd 下的真实路径解析
- 历史 `~/.remote-claude/*.log` 主日志路径的残留调用点

修复原则：

- 主日志目录统一收敛到 `/tmp/remote-claude/`。
- 调试日志、socket、运行时配置、用户配置各自职责分离，不混用路径语义。
- 路径处理优先复用已有真相源，避免各文件重复拼接字符串。

#### 4. 异常处理

重点补足边界：

- 网络 I/O
- socket / WebSocket 关闭
- 日志文件初始化与轮转挂载
- `/tmp/remote-claude/` 不可写、目录不存在或创建失败
- 启动期到运行期日志切换过程

修复原则：

- 仅在系统边界上捕获异常。
- 不增加无意义的广域 `except Exception: pass`。
- 出错后优先保证：状态一致、用户有提示、日志有记录。

#### 5. 保活处理

重点检查：

- WebSocket `ping_interval` / `ping_timeout` 设置是否合理
- 断线后原因是否能透传到用户可见层
- 是否存在因读写循环退出顺序问题导致的僵尸连接或无提示退出

修复原则：

- 以现有 WebSocket ping/pong 能力为主，只做必要修正。
- 不引入额外应用层心跳协议。
- 让保活异常最终归一到统一断线原因链路。

### 三、统一日志初始化模块设计

#### 目标

新增一个统一日志初始化模块，负责：

- 创建 `/tmp/remote-claude/`
- 为不同角色返回对应 logger / handler 配置
- 挂载 `RotatingFileHandler`
- 保证 handler 幂等，避免重复初始化导致重复写日志
- 统一 formatter、level、编码与轮转参数

#### 角色划分

主日志按角色拆分为三个固定文件：

- `/tmp/remote-claude/client.log`
- `/tmp/remote-claude/server.log`
- `/tmp/remote-claude/lark.log`

角色定义：

- **client**：本地客户端、远程客户端、控制命令客户端侧通用日志
- **server**：PTY 代理服务、socket / WebSocket 服务端、启动/运行期服务端主日志
- **lark**：飞书客户端入口、轮询、卡片服务、桥接链路日志

#### 轮转策略

统一采用按大小轮转：

- `maxBytes = 10 * 1024 * 1024`
- `backupCount = 5`

理由：

- 对低频命令与长驻服务都更稳定。
- 避免时间轮转在测试环境和低活跃环境下产生不可预测差异。
- 便于测试时仅断言 handler 类型与关键参数。

#### 幂等初始化要求

统一日志初始化模块必须满足：

- 同一角色多次调用不会重复添加 handler。
- 能区分主日志 handler 与已有会话级调试 handler。
- 若日志目录不可创建，调用方至少能回退到 stderr / 现有控制台日志，而不是静默失败。

### 四、现有日志体系的收敛规则

#### 1. 主日志收敛

需要把当前作为主日志用途的路径从 `~/.remote-claude/*.log` 收敛到 `/tmp/remote-claude/*.log`，重点包括：

- startup / runtime server 主日志
- server stderr 重定向目标
- lark 主日志入口
- client 侧尚未统一的日志初始化点

#### 2. 调试日志保留

以下类型保持原有“会话级 / 调试级”定位，不强行并入主日志：

- `_screen.log`
- `_pty_raw.log`
- 会话级 debug 日志
- 为特定会话或诊断场景生成的辅助文件

原则是：

- **主日志**解决“系统在做什么、哪一层失败了”。
- **调试日志**解决“这个会话里的字节流 / 终端快照到底长什么样”。

#### 3. 日志内容原则

日志字段尽量结构化，至少覆盖：

- `stage`
- `session`（如适用）
- `host` / `port`（如适用）
- `reason` 或 `error`

并遵守：

- token 必须脱敏
- 不把完整堆栈直接输出到用户终端
- 终端提示简洁，详细信息写日志

### 五、用户可见行为设计

#### 1. 终端错误输出

用户侧只看到简洁错误，例如：

- `✗ 连接失败: <原因>`
- `✗ 发送失败: <原因>`
- `✗ 远程连接已关闭: <原因>`
- `✗ 日志初始化失败，已回退到标准错误输出`

不直接打印冗长 traceback。

#### 2. 帮助与参数行为

- `--help/-h` 路径不触发真实连接或副作用。
- 关键远程命令在缺参时报清晰错误。
- `lark` 这类子命令树在缺少子命令时给出帮助或明确错误，不出现沉默失败。

#### 3. 日志落点告知

除非当前错误与日志初始化直接相关，否则不在每次命令成功时额外冗长提示日志路径。仅在失败或排障场景下提示主日志位于 `/tmp/remote-claude/`。

### 六、测试策略

#### 1. 稳定性修复测试

补充或调整测试，覆盖：

- WebSocket 建连失败、发送失败、读取关闭、异常关闭
- BaseClient 对断线原因的透传与用户输出
- 远程参数校验与帮助模式
- bin / scripts 路径解析的稳定性（如已有测试则按需补充，不重复覆盖）

#### 2. 日志初始化测试

补充测试验证：

- 统一日志初始化模块为不同角色返回正确日志文件
- 使用 `RotatingFileHandler`
- 轮转参数为 `10MB × 5`
- 重复初始化不会重复挂载 handler
- 日志目录为 `/tmp/remote-claude/`

#### 3. 回归测试

优先复用或扩展以下现有测试文件：

- `tests/test_client_integration.py`
- `tests/test_cli_help_and_remote.py`
- `tests/test_entry_lazy_init.py`
- `tests/test_list_display.py`

如需新增测试，应只为本次新增行为建立最小测试文件，不扩散测试面。

#### 4. 文档同步

更新：

- `CLAUDE.md`：补充主日志路径与角色拆分说明
- `tests/TEST_PLAN.md`：补充主日志分流与轮转、稳定性扫描相关验证项

### 七、风险与控制

#### 风险 1：日志收敛影响现有启动期排障

控制：

- 保留启动期到运行期的阶段切换语义。
- 先收敛主日志目标路径，再最小化调整启动链路。
- 调试日志能力保持不动。

#### 风险 2：统一日志初始化后出现重复日志

控制：

- 给主日志 handler 打标记。
- 初始化函数做幂等检查。
- 测试覆盖重复初始化场景。

#### 风险 3：参数/连接修复引入行为回退

控制：

- 只在明确问题边界收敛逻辑。
- 保持现有命令写法兼容。
- 为关键命令路径补回归测试。

#### 风险 4：`/tmp` 路径在异常环境下不可写

控制：

- 初始化失败时回退到 stderr。
- 日志中明确记录日志初始化失败原因。
- 不因日志系统失败阻断核心命令主流程，除非该流程明确依赖日志文件本身。

## 验收标准

满足以下条件即认为本次设计达成：

1. `server/`、`client/`、`bin/`、`remote_claude.py` 中与连接、参数、路径、异常、保活相关的已识别真实问题被修复。
2. 客户端 / 服务端 / lark 三类主日志统一落到 `/tmp/remote-claude/`。
3. 主日志全部使用按大小轮转，参数为 `10MB × 5`。
4. 现有会话级调试日志能力仍可用，没有被错误合并或删除。
5. 关键错误路径用户可看到简洁原因，详细信息进入对应主日志。
6. 相关测试通过，并且 `CLAUDE.md` 与 `tests/TEST_PLAN.md` 已同步更新。
